# Tech Stack Map

## Repository Shape
- The repo is a mixed hardware/software project centered on MobileNet v1 inference.
- Python model code lives in `src/`.
- Python tooling and dataset/quantization utilities live in `scripts/`.
- Hardware RTL is implemented in `rtl/` with testbenches in `rtl/tb/`.
- PS-side bring-up helpers for Xilinx flows live in `sw/`.
- Generated or checked-in artifacts appear under `rtl/mem/`, `quantized_models/`, `obj_dir/`, `processed_frames/`, and `selected_frames/`.

## Primary Languages
- Python is the main scripting and model-development language across `src/*.py` and `scripts/*.py`.
- SystemVerilog is the hardware implementation language across `rtl/common/*.sv`, `rtl/blocks/*.sv`, and `rtl/tb/*.sv`.
- C is used for software bring-up examples in `sw/*.c`.
- C headers in `sw/*.h` define the control/register interface used by software against the RTL wrappers.

## Python Runtime And Packaging
- `environment.yml` defines a Conda environment named `mobilenet_env`.
- `environment.yml` pins `python=3.11` and installs dependencies from `requirements.txt`.
- `requirements.txt` is the only declared Python dependency manifest; there is no `pyproject.toml`, `setup.py`, or `package.json`.
- There is no lockfile for Conda or pip, so the Python environment is only loosely reproducible.
- `src/__init__.py` exists, but this is not packaged as an installable library; scripts instead add the repo root to `sys.path` directly in files such as `scripts/export_mobilenet_int8_mem.py` and `scripts/quantize_from_imagenet_val.py`.

## Declared Python Dependencies
- `requirements.txt` declares `numpy`, `tensorflow`, `tensorflow-datasets`, `jupyter`, `matplotlib`, `pytest`, `opencv-python`, and `transparent_background`.
- The declared workflow assumes Conda invocation from the repo root, as documented in `scripts/README.md`.
- `jupyter` is declared, but there are no notebook files in the repo.
- `pytest` is declared, but there are no visible Python test modules or `def test_...` functions in tracked source files.

## Python Dependencies Used In Code
- Numerical kernels and array manipulation rely on `numpy` throughout `src/` and `scripts/`.
- Keras/TensorFlow drive model definition, weight loading, preprocessing, TFLite conversion, and TFLite inference in `src/MobileNet_tf.py`, `scripts/quantize_from_imagenet_val.py`, `scripts/eval_imagenet_val.py`, and related scripts.
- `tensorflow_datasets` is used in `scripts/build_imagenet_npz.py` to materialize ImageNet-style datasets.
- `matplotlib` is used for visualization in `scripts/image_crop.py`.
- OpenCV is used in `scripts/object_crop.py` via `cv2` for image/video IO and mask handling.
- `transparent_background.Remover` is used in `scripts/object_crop.py` for foreground extraction.
- Pillow is used in many scripts via `from PIL import Image, ImageDraw`, including `scripts/image_to_input_mem.py`, `scripts/run_roi_golden.py`, and `scripts/roi_visualize.py`.
- Pillow is not declared in `requirements.txt`, so the effective runtime dependencies are broader than the manifest.

## Python Model Stack
- `src/MobileNet_tf.py` defines a TensorFlow/Keras MobileNet v1 variant via the Functional API.
- The TensorFlow model uses `Conv2D`, `DepthwiseConv2D`, `BatchNormalization`, `ReLU(max_value=6.0)`, `GlobalAveragePooling2D`, `Dropout`, and a 1x1 `Conv2D` classification head.
- `src/MobileNet_tf.py` also exports weights into a NumPy-friendly parameter dictionary with `export_weights_to_numpy_dict`.
- `src/MobileNet_np.py` implements an inference-only NumPy MobileNet v1 forward path using helper modules in `src/conv.py`, `src/depthwise_conv.py`, `src/batchnorm.py`, `src/pooling.py`, `src/Fully_connected.py`, and `src/softmax.py`.
- The TensorFlow and NumPy paths share the same layer schedule through `DEPTHWISE_BLOCK_SPECS` in both `src/MobileNet_tf.py` and `src/MobileNet_np.py`.

## Model And Data Artifacts
- Keras weights are expected at `mobilenet_imagenet.weights.h5`, created by `scripts/add_weights.py`.
- Quantized TFLite models are stored in `quantized_models/`, including `quantized_models/mobilenet_int8_ilsvrc2012_5000.tflite`.
- RTL parameter memories are stored as hex `.mem` files in `rtl/mem/`, generated by `scripts/export_mobilenet_int8_mem.py`.
- Golden outputs and hardware dumps also land in `rtl/mem/`, including `fc_expected.mem`, `fc_logits_expected.mem`, `fc_out_hw.mem`, and per-layer dumps like `layer1_out_hw.mem`.
- ROI and background-removal pipelines emit assets under `processed_frames/`, `selected_frames/`, and `rtl/mem/`.

## Hardware Stack
- The accelerator is written in SystemVerilog and targets an int8 MobileNet v1 datapath, described in `rtl/README.md`.
- Reusable datapath components live in `rtl/common/`, including `line_buffer_3x3.sv`, `dw_conv_3x3.sv`, `conv3x3_mac_vec.sv`, `requant_q31.sv`, and `dual_port_ram_async.sv`.
- Higher-level blocks live in `rtl/blocks/`, including `mobilenet_v1_top.sv`, `mobilenet_v1_ctrl.sv`, `mobilenet_v1_param_cache.sv`, `mobilenet_v1_axi_lite.sv`, `mobilenet_v1_axi_ctrl_bram.sv`, `mobilenet_v1_htg_vsl5_top.sv`, and `mobilenet_v1_vivado_bram_top.sv`.
- Hardware parameter memories are loaded through `$readmemh` using `INIT_*` file parameters in modules such as `rtl/blocks/mobilenet_v1_top.sv` and `rtl/blocks/mobilenet_v1_param_cache.sv`.
- The design supports an optional Xilinx XPM-backed RAM path through `XILINX_XPM` and `USE_XPM_RAM`, especially in `rtl/common/dual_port_ram_async.sv` and the board-facing wrapper modules.

## Hardware Verification And Build Signals
- Testbenches live in `rtl/tb/`, including `tb_mobilenet_v1_top.sv`, `tb_mobilenet_v1_axi_lite.sv`, `tb_mobilenet_v1_axi_ctrl_bram.sv`, `tb_mobilenet_v1_vivado_bram_top.sv`, `tb_param_cache.sv`, and `tb_line_buffer_3x3.sv`.
- Checked-in `obj_dir/*.mk` files such as `obj_dir/Vtb_mobilenet_v1_top.mk` and `obj_dir/Vtb_param_cache.mk` indicate Verilator-generated build outputs are part of the current workflow.
- There is no top-level `Makefile` or shell harness in the repo root, so the simulation/build entrypoint is implicit rather than standardized.
- `rtl/tb/tb_mobilenet_v1_top.sv` wires the DUT against generated mem files in `rtl/mem/`, so simulation depends on preprocessing/export scripts having run first.

## Embedded Software Stack
- `sw/mobilenet_v1_regs.h` and `sw/mobilenet_v1_ctrl_bram_regs.h` provide register helpers for the AXI-Lite control plane.
- `sw/mobilenet_v1_ctrl_bram_xil.h` provides Vitis-style helpers using `Xil_In32` and `Xil_Out32`.
- `sw/mobilenet_v1_bringup_example.c` and `sw/mobilenet_v1_vitis_bringup_example.c` are bare-metal bring-up examples rather than a full application.
- The software assumes Xilinx-generated `xparameters.h` may exist but includes fallback base addresses when it does not.

## Configuration And Environment Conventions
- There are no `.env` files, no secrets manager hooks, and no application server configuration.
- Runtime configuration is mostly CLI-driven through `argparse` in almost every script under `scripts/`.
- Common default paths are hardcoded, for example `ILSVRC2012_val`, `ILSVRC2012_devkit_t12/...`, `quantized_models/...`, and `rtl/mem/...`.
- `scripts/README.md` is the closest thing to an execution guide; it documents expected commands and default file locations.
- `.transparent-background/config.yaml` and `.transparent-background/ckpt_base.pth` show that one toolchain caches third-party model assets inside the repo workspace.

## Practical Execution Flow
- Build or load MobileNet weights with `scripts/add_weights.py`.
- Quantize the Keras model to TFLite with `scripts/quantize_from_imagenet_val.py`.
- Export TFLite or fused Keras weights into RTL `.mem` files with `scripts/export_mobilenet_int8_mem.py`.
- Convert images into accelerator input memories with `scripts/image_to_input_mem.py` or `scripts/image_to_input_mem_roi.py`.
- Run RTL testbenches against the generated mem files in `rtl/tb/`.
- Compare hardware outputs against golden references with scripts such as `scripts/compare_fc.py`, `scripts/compare_fc_logits.py`, and `scripts/compare_all_layers.py`.

## Stack Gaps Worth Knowing
- Dependency declaration is incomplete because Pillow is used but not listed in `requirements.txt`.
- There is no single reproducible build/test command across Python, RTL simulation, and Vitis bring-up.
- The repo mixes source code and large generated artifacts in-place, so planning future automation will need clearer artifact boundaries.
